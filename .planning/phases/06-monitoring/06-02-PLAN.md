- `object CheckCommand "docker-stack"`:

  <name>Task 2: Create monitoring integration README</name>

  <action>
Create `monitoring/README.md` — a self-contained integration guide for the Icinga2 master admin.

**Structure:**

1. **Title and Overview**
   - "Odoo Production Monitoring — Icinga2 Integration Guide"
   - Brief: two custom check plugins (Docker + PostgreSQL) and service definition templates for integrating the Odoo host with an existing Icinga2 master

2. **Architecture Overview**
   - Bullet list explaining: master schedules checks via command_endpoint, agent executes plugins locally, plugins query Docker daemon and PostgreSQL via docker exec
   - ASCII diagram showing: `Icinga2 Master → (TLS) → Icinga2 Agent → check_docker_stack → Docker daemon` and `Icinga2 Agent → check_postgres_health → docker exec → PostgreSQL`

3. **Prerequisites**
   - Icinga2 agent installed and registered with master (MON-01 — handled by Icinga2 master project)
   - Standard system checks deployed (MON-04 — handled by Icinga2 master project, includes check_cpu, check_mem, check_disk, check_load)
   - `nagios` user added to `docker` group: `sudo usermod -aG docker nagios` (required for plugins to access Docker socket)
   - Docker CE running on the Odoo host
   - Note: the agent installation and system resource checks are NOT part of this package — they are deployed by the Icinga2 master project's agent provisioning workflow

4. **Directory Structure**
   - Show the `monitoring/` directory tree with descriptions of each file

5. **Plugin Deployment (on Odoo agent host)**
   - Copy plugins to `/usr/lib/nagios/plugins/custom/`:
     ```
     sudo mkdir -p /usr/lib/nagios/plugins/custom
     sudo cp monitoring/plugins/check_docker_stack /usr/lib/nagios/plugins/custom/
     sudo cp monitoring/plugins/check_postgres_health /usr/lib/nagios/plugins/custom/
     sudo chmod 755 /usr/lib/nagios/plugins/custom/check_*
     sudo chown root:root /usr/lib/nagios/plugins/custom/check_*
     ```
   - Verify with: `sudo -u nagios /usr/lib/nagios/plugins/custom/check_docker_stack --help`
   - If permission denied: `sudo usermod -aG docker nagios && sudo systemctl restart icinga2`

6. **Service Definition Deployment (on Icinga2 master)**
   - Step-by-step:
     1. Copy `commands.conf` to `/etc/icinga2/zones.d/global-templates/odoo-commands.conf`
     2. Copy `services.conf` to `/etc/icinga2/zones.d/master/odoo-services.conf`
     3. Copy `notifications.conf` to `/etc/icinga2/zones.d/master/odoo-notifications.conf`
     4. Edit placeholders (list each PLACEHOLDER and what to replace with)
     5. Ensure the host object has `vars.odoo_host = true` and `vars.agent_endpoint = "hostname"`
     6. Validate: `icinga2 daemon -C`
     7. Reload: `sudo systemctl reload icinga2`

7. **Check Details**
   - Table for each check: name, what it monitors, default thresholds, configurable arguments
   - Docker check: containers monitored, warning/critical thresholds for restarts, health status evaluation
   - PostgreSQL check: 4 metrics with thresholds, note about docker exec latency overhead (~20-50ms)

8. **Threshold Tuning**
   - Explain how to override thresholds via Icinga2 custom vars on the host or service object
   - Example: `vars.postgres_health_warn_conn = 25` to lower connection warning threshold

9. **Troubleshooting**
   - Table format: Symptom | Cause | Fix
   - Common issues: permission denied (docker group), container not found (wrong name), UNKNOWN status (plugin path wrong), perfdata not graphing (label format)

10. **Performance Data**
    - List all perfdata labels output by each plugin with units and ranges
    - Note compatibility with Graphite, InfluxDB, PNP4Nagios via standard Nagios perfdata format

Do NOT include emojis. Use standard Markdown. Keep it direct and actionable — this is a reference doc for an experienced Icinga2 admin.
  </action>
