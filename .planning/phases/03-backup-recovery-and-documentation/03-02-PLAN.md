**docs/architecture.md (DOC-01):**

Create a comprehensive architecture overview document. Dual audience: concise enough for experienced sysadmins to scan quickly, thorough enough for contractors/MSPs with zero context on this stack.

**Sections:**

1. **System Overview** — One-paragraph summary: Odoo Community 19 on a single DigitalOcean droplet, containerized with Docker Compose, behind host-installed Nginx with Let's Encrypt SSL.

2. **Network Topology — ASCII Diagram** — Terminal-friendly diagram showing:
   - Internet -> DO Cloud Firewall (ports 80, 443, 9292) -> Droplet
   - On droplet: Nginx (host, ports 80/443) -> Odoo container (127.0.0.1:8069/8072) -> PostgreSQL container (backend network)
   - Docker networks: odoo-frontend (bridge), odoo-backend (internal)
   - Block Storage Volume: /mnt/odoo-prod-data/ with postgres-data/, odoo-filestore/, backups/
   - DO Spaces: odoo-prod-tfstate (Standard), odoo-prod-backups (Cold Storage)

3. **Network Topology — Mermaid Diagram** — Same topology in Mermaid format for GitHub rendering. Use a flowchart (graph TD) showing the same components and connections.

4. **Component Table** — Table with columns: Component, Location, Port(s), Purpose, Key Config
   - Nginx (host, 80/443, reverse proxy, /etc/nginx/sites-available/odoo)
   - Odoo (container, 127.0.0.1:8069/8072, application, /opt/odoo/odoo.conf)
   - PostgreSQL (container, backend-only/5432, database, /opt/odoo/postgresql.conf)
   - Docker (host, -, container runtime, /etc/docker/daemon.json)
   - UFW (host, -, firewall, rules for 80/443/9292)
   - fail2ban (host, -, intrusion prevention, SSH + Odoo jails)
   - certbot (host, -, SSL renewal, systemd timer)
   - auditd (host, -, PCI-DSS logging, /etc/audit/rules.d/)

5. **Data Flow** — Describe the request flow: client HTTPS -> Nginx TLS termination -> proxy_pass to Odoo 8069 -> Odoo queries PostgreSQL via Docker backend network -> response back up the chain. Longpolling on 8072. All persistent data on Block Storage Volume.

6. **Security Architecture** — UFW as single firewall (Docker iptables:false), SSH key-only on port 9292, fail2ban jails, non-root containers, internal backend network, HSTS headers, database manager blocking.

7. **Backup Architecture** — Daily pg_dump + filestore tar to Block Storage, rclone to DO Spaces Cold Storage, 7/4 local retention, 30-day remote retention. Reference docs/operations.md for procedures.

8. **Infrastructure Diagram** — Show DO resources: VPC, droplet, Block Storage Volume, cloud firewall, Spaces buckets (tfstate + backups).

**docs/enterprise-migration.md (DOC-04):**

Create a step-by-step Enterprise edition migration guide. Reference: https://www.odoo.com/documentation/19.0/administration/on_premise/community_to_enterprise.html

**Sections:**

1. **Prerequisites** — Odoo Enterprise subscription, enterprise addons source (registry or download), current system backup verified.

2. **Pre-Migration Backup** — Run full backup (reference 06-backup-daily.sh), verify backup with 08-restore-backup.sh --verify-only, confirm backup in DO Spaces.

3. **Migration Steps:**
   a. Stop Odoo container
   b. Download/obtain enterprise addons for Odoo 19
   c. Update docker-compose.yml: add bind mount for enterprise addons (`./enterprise:/mnt/extra-addons`)
   d. Update odoo.conf: add enterprise addons path to `addons_path`
   e. Start Odoo container
   f. Install `web_enterprise` module via Odoo UI or CLI
   g. Activate license via Settings -> Odoo Enterprise License
   h. Verify enterprise features active (Settings should show Enterprise branding)

4. **Verification Checklist** — Enterprise dashboard visible, all existing CRM/Project data intact, user logins working, SSL still valid.

5. **Rollback Procedure** — Stop Odoo, remove enterprise bind mount from docker-compose.yml, revert odoo.conf addons_path, start Odoo, verify Community edition functions. If data was modified by Enterprise modules: restore from pre-migration backup.

6. **Notes** — Enterprise Docker image alternative (private registry), version compatibility requirements, support contacts.

**docs/deployment-runbook.md (DOC-02):**

Create an end-to-end deployment guide that takes a new operator from a fresh git clone to a running Odoo instance in production. Dual audience per user decision.

**Sections:**

1. **Prerequisites** — What's needed before starting:
   - DigitalOcean account with API token
   - DO Spaces access keys (separate from API token)
   - Domain name with ability to manage DNS
   - SSH key pair (ed25519 recommended)
   - Local machine with: Terraform >= 1.6.3, SSH client
   - SMTP credentials for backup failure alerts (can be configured post-deploy)
   - Estimated time: ~45 minutes for full deployment

2. **Step 1: Clone and Configure** — `git clone`, copy tfvars.example and .env.example, fill in values. Explain each variable briefly. Reference password rules from .env.example.

3. **Step 2: Provision Infrastructure (Terraform):**
   - Set environment variables: DIGITALOCEAN_TOKEN, AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY
   - `cd infra && terraform init && terraform plan && terraform apply`
   - Note the outputs: droplet IP, volume path
   - Set up DNS: A record pointing domain to droplet IP

4. **Step 3: Copy Files to Droplet:**
   - `scp -r config/ scripts/ root@<IP>:/tmp/odoo-setup/`
   - `scp config/.env root@<IP>:/tmp/odoo-setup/config/` (the actual .env with real secrets)

5. **Step 4: Harden Host (Script 01):**
   - `ssh root@<IP>` then `bash /tmp/odoo-setup/scripts/01-harden-host.sh`
   - Note: after this, reconnect on port 9292 as deploy user
   - `ssh -p 9292 deploy@<IP>`

6. **Step 5: Install Docker (Script 02):**
   - `sudo bash /tmp/odoo-setup/scripts/02-install-docker.sh`

7. **Step 6: Deploy Stack (Script 03):**
   - `sudo bash /tmp/odoo-setup/scripts/03-deploy-stack.sh`
   - Verify containers running: `docker compose -f /opt/odoo/docker-compose.yml ps`

8. **Step 7: Setup Nginx + SSL (Script 04):**
   - `sudo bash /tmp/odoo-setup/scripts/04-setup-nginx.sh <domain> <email>`
   - Verify: `curl -I https://<domain>` should return 200 with HSTS header

9. **Step 8: Setup Backups (Script 05):**
   - `sudo bash /tmp/odoo-setup/scripts/05-setup-backups.sh`
   - Verify: check cron installed, test manual backup run

10. **Step 9: Verify Deployment:**
    - Access https://<domain> in browser
    - Log in with admin credentials
    - Verify CRM and Project modules accessible
    - Run manual backup and verify-only restore

11. **Troubleshooting** — Common issues:
    - Terraform apply fails (token permissions, Spaces access)
    - SSH connection refused after hardening (wrong port, firewall)
    - Certbot fails (DNS not propagated, rate limits)
    - Containers not starting (disk space, .env errors)
    - Odoo shows 502 (container not ready, wrong port binding)

**docs/operations.md (DOC-03):**

Create operational procedures for day-to-day management. Each procedure should be self-contained — operator can jump to any section.

**Sections:**

1. **Backup Operations:**
   - How automated backups work (cron schedule, what runs when)
   - Manual backup trigger: `sudo /opt/odoo/scripts/06-backup-daily.sh`
   - Check backup status: `cat /opt/odoo/backup-status.json`
   - List local backups: `ls -la /mnt/odoo-prod-data/backups/daily/`
   - List remote backups: `rclone ls --config /opt/odoo/rclone.conf spaces:odoo-prod-backups/`

2. **Restore Operations:**
   - Verify a backup (non-destructive): `sudo bash /opt/odoo/scripts/08-restore-backup.sh --verify-only`
   - Restore specific local backup: `sudo bash /opt/odoo/scripts/08-restore-backup.sh --production --file /path/to/dump`
   - Restore from DO Spaces: `sudo bash /opt/odoo/scripts/08-restore-backup.sh --production --from-spaces --date 2026-03-15`
   - Post-restore verification steps

3. **Odoo Version Update:**
   - Pre-update: run full backup, verify backup
   - Pull new image: `docker pull odoo:19` (or specific tag)
   - Recreate container: `cd /opt/odoo && docker compose up -d odoo`
   - Verify: check Odoo version in Settings -> About, test key workflows
   - Rollback: `docker compose stop odoo`, restore backup, `docker compose up -d odoo`

4. **PostgreSQL Major Version Upgrade:**
   - Warning: major version upgrades require pg_dump/pg_restore cycle
   - Backup current database
   - Update image tag in docker-compose.yml (e.g., postgres:19)
   - Stop containers, remove PG data directory, start with new image
   - Restore from backup
   - Verify

5. **Droplet Resize (Vertical Scaling):**
   - Option A: Via Terraform — update droplet_size in tfvars, `terraform apply`
   - Option B: Via DO Console — power off droplet, resize, power on
   - After resize: update Odoo worker count in odoo.conf if CPU increased, restart container
   - Reference worker sizing guidance from Odoo docs

6. **Volume Resize (Storage Scaling):**
   - Via Terraform: update volume_size in tfvars, `terraform apply`
   - After resize: `resize2fs /dev/disk/by-id/<volume-id>` on droplet to expand filesystem
   - Verify: `df -h /mnt/odoo-prod-data/`

7. **SSL Certificate Issues:**
   - Check cert expiry: `sudo certbot certificates`
   - Force renewal: `sudo certbot renew --force-renewal`
   - Check renewal timer: `sudo systemctl status certbot.timer`
   - Full reissue: re-run relevant section of 04-setup-nginx.sh

8. **Log Locations:**
   - Odoo: `docker logs odoo-app` or `docker logs --tail 100 -f odoo-app`
   - PostgreSQL: `docker logs odoo-db`
   - Nginx: `/var/log/nginx/odoo-access.log`, `/var/log/nginx/odoo-error.log`
   - Backup: `/var/log/odoo-backup.log`, `/var/log/odoo-backup-sync.log`
   - SSH/auth: `journalctl -u ssh`
   - fail2ban: `sudo fail2ban-client status`, `sudo fail2ban-client status sshd`
   - System: `journalctl -f`, `dmesg`

9. **Emergency Procedures:**
   - Odoo down: `docker compose -f /opt/odoo/docker-compose.yml restart odoo`
   - Database connection issues: check PG health `docker exec odoo-db pg_isready`, restart if needed
   - Disk full: check with `df -h`, clean old backups, check Docker logs rotation
   - Locked out of SSH: access via DO console, check fail2ban, check UFW rules
