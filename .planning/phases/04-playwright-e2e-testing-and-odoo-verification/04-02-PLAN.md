    3. "database manager is blocked" — navigate to `/web/database/manager`, verify either HTTP 403 or a redirect away (per ODOO-03 and PROXY-04). If running against production, expect 403 from Nginx. If local, Odoo may redirect — check that the page does NOT show the database manager form.

    1. "CRM module is installed" — use `adminPage`, create SettingsPage, check `isModuleInstalled('CRM')`

    3. "CRM app is accessible from menu" — use AppMenuPage to verify CRM appears in the app menu
    4. "Project app is accessible from menu" — same for Project

    Key test patterns:
    - Use `test.describe` to group related tests
    - Each test gets its own `page` (via Playwright's default isolation)
    - Admin-authenticated tests use `adminPage` from auth fixture
    - Smoke tests should be fast (< 30s each)

    Create two workflow test files. These tests CREATE DATA and should run against local staging only (though they're not in the `setup/` directory, so they're not auto-excluded from production — add a test.skip condition if `PROD_ODOO_URL` matches the baseURL).

    **tests/workflows/crm-lead.spec.ts**: Import `test` from auth fixture for `adminPage`. Full CRM lead lifecycle:
    1. "create a new CRM lead" — use CRMPage.createLead with a unique name (include timestamp to avoid collisions), verify lead appears in the pipeline
    2. "advance lead through stages" — open the created lead, change stage using CRMPage.setStage (e.g., "Qualified" then "Proposition"), verify stage changes are reflected
    3. "mark lead as won" — call CRMPage.markWon(), verify the lead is in the Won stage or shows a won indicator
    4. "create and mark a lead as lost" — create a second lead, advance it, then call CRMPage.markLost(), verify lost state

    Use `test.describe.serial()` so tests run in order (each depends on previous state). Use unique lead names with `Date.now()` suffix.

    **tests/workflows/project-task.spec.ts**: Import `test` from auth fixture for `adminPage`. Project + task management:
    1. "create a new project" — use ProjectPage.createProject with unique name, verify project appears
    2. "add tasks to the project" — create 2-3 tasks with different names, verify they appear in the project's task list
    3. "change task stage" — open a task, change its stage (e.g., "In Progress" then "Done"), verify stage reflects
    4. "tasks persist after page reload" — reload the project page, verify tasks still exist

    Use `test.describe.serial()` for ordered execution.

    Key patterns:
    - Generate unique names: `const leadName = \`Test Lead \${Date.now()}\``
    - After each Odoo form save, wait for the success notification or form to exit edit mode
    - Use `page.waitForResponse()` after stage changes to wait for the RPC to complete
    - Add `test.slow()` on longer workflow tests if needed (doubles the timeout)
    - Clean up created data in `test.afterAll` if feasible (delete test leads/projects), but don't fail the suite if cleanup fails

    - Project test covers create project → add tasks → change status → persist check
