  <name>Task 1: Scaffold OdooKit project with Playwright, Docker Compose, and helpers</name>
  <files>
    odookit/package.json

  <action>
    Create `odookit/` directory with the following files:

    **package.json**: Name `odookit`, private, ES module type. Dependencies: `@playwright/test` ^1.58, `typescript` ^5.x, `@types/node` ^22.x, `dotenv` ^16.x. Scripts:
    - `test:local` — `npx playwright test --project=local`
    - `test:prod` — `npx playwright test --project=production`
    - `test:smoke` — `npx playwright test --project=local tests/smoke/`
    - `setup:local` — `npx playwright test --project=local tests/setup/ && echo "UAT Ready"` (will be enhanced in Plan 03)
    - `teardown:local` — `docker compose down -v`

    **tsconfig.json**: Target ES2022, module NodeNext, moduleResolution NodeNext, strict true, outDir dist, rootDir `.`, include `["**/*.ts"]`, exclude `["node_modules", "dist"]`.

    **playwright.config.ts**: Use the exact pattern from 04-RESEARCH.md Pattern 2:
    - Load dotenv at top
    - `testDir: './tests'`, `timeout: 60_000`, `expect.timeout: 10_000`
    - `fullyParallel: false` (Odoo has shared state)
    - `retries: 1`
    - Reporter: check `process.env.REPORT` — if set use `[['html', { open: 'never', outputFolder: 'reports' }]]`, else `[['list']]`
    - Global `use`: `screenshot: 'only-on-failure'`, `video: 'retain-on-failure'`, `trace: 'retain-on-failure'`
    - Two projects:
      1. `local` — baseURL from `LOCAL_ODOO_URL` env var (default `http://localhost:8069`), Desktop Chrome, testIgnore empty
      2. `production` — baseURL from `PROD_ODOO_URL` env var, Desktop Chrome, `testIgnore: ['**/setup/**']` to block destructive tests on prod

    **.env.example**: Template with all variables from research (LOCAL_ODOO_URL, PROD_ODOO_URL, ADMIN_LOGIN, ADMIN_PASSWORD, TEST_USER_LOGIN, TEST_USER_PASSWORD, LOCAL_POSTGRES_USER, LOCAL_POSTGRES_PASSWORD, LOCAL_POSTGRES_DB, INFRA_SSH_HOST, INFRA_SSH_PORT, INFRA_SSH_USER, REPORT). Add comment header: "OdooKit — Odoo automation and verification toolkit". Include warnings about unsafe characters in passwords ($, backticks, quotes — per project memory feedback_env_password_rules.md).

    **.gitignore**: node_modules/, dist/, reports/, .env, test-results/, playwright-report/, blob-report/.

    **docker-compose.yml**: Local staging stack from research Pattern 4. Odoo 19 + PostgreSQL 18 with health checks. Use env vars with defaults (LOCAL_POSTGRES_USER:-odoo, etc). Odoo ports 8069:8069 and 8072:8072. PostgreSQL health check uses pg_isready. Odoo health check curls /web/health. No resource limits (local dev).

    **helpers/env.ts**: Export a function `loadEnv()` that calls `dotenv.config()` and returns a typed object with all env vars. Throw descriptive errors if required vars (ADMIN_LOGIN, ADMIN_PASSWORD) are missing. Optional vars (PROD_ODOO_URL, TEST_USER_LOGIN, etc.) return undefined if unset.

    **helpers/wait-for-odoo.ts**: Export `waitForOdoo(url: string, timeoutMs = 120_000)` that polls `${url}/web/health` every 2 seconds until HTTP 200 or timeout. Uses native `fetch`. Logs progress to console.

    **helpers/docker-compose.ts**: Export `composeUp()` and `composeDown(removeVolumes = false)` that run `docker compose` commands via `child_process.execSync` in the odookit directory. `composeUp` calls `docker compose up -d --wait` and then `waitForOdoo('http://localhost:8069')`. `composeDown` calls `docker compose down` with optional `-v` flag.

    Run `cd odookit && npm install && npx playwright install chromium --with-deps` to install everything.
  </action>

  <action>
    Create Page Object Models following Playwright POM pattern (class with Page in constructor, locators as readonly properties, methods for user actions).

    **pages/login.page.ts**: From research Pattern 1. Selectors: `input[name="login"]`, `input[name="password"]`, `.oe_login_form button[type="submit"]`. Methods: `goto()` navigates to `/web/login`, `login(email, password)` fills and submits then waits for `.o_main_navbar`. Add `isLoggedIn()` that checks if `.o_main_navbar` is visible.

    **pages/app-menu.page.ts**: Encapsulates the Odoo app launcher. Method `openApp(appName: string)` — clicks the app in the home menu or navbar. Use text-based locator `page.getByRole('menuitem', { name: appName })` as primary, fall back to `[data-menu-xmlid]` selectors. After clicking, wait for `.o_action_manager .o_view_controller` to be visible. Method `isAppInstalled(appName: string)` — navigates to home menu and checks if the app tile exists.

    **pages/crm.page.ts**: CRM module interactions. Methods:
    - `openCRM()` — uses AppMenuPage.openApp('CRM')
    - `createLead(name: string, options?: { contactName?: string, email?: string })` — clicks "New", fills the form (input for lead name, optional contact/email fields), saves. Wait for onchange responses between field fills.
    - `openLead(name: string)` — finds and clicks the lead in list/kanban view
    - `setStage(stageName: string)` — in the lead form, changes the stage via the status bar (`page.locator('.o_statusbar_status button', { hasText: stageName }).click()`). Wait for save.
    - `getLeadStage()` — returns the current active stage text from the status bar
    - `markWon()` / `markLost()` — clicks Won/Lost buttons in the lead form

    **pages/project.page.ts**: Project module interactions. Methods:
    - `openProject()` — uses AppMenuPage.openApp('Project')
    - `createProject(name: string)` — clicks "New" button, fills project name, confirms
    - `openProject(name: string)` — clicks the project from list/kanban
    - `createTask(projectName: string, taskName: string, options?: { assignee?: string })` — opens project, creates a new task with optional assignment
    - `openTask(taskName: string)` — clicks the task from kanban/list
    - `setTaskStage(stageName: string)` — changes task stage via status bar or kanban column

    **pages/settings.page.ts**: Odoo Settings page. Methods:
    - `openSettings()` — navigates to Settings app
    - `isDatabaseManagerDisabled()` — checks that the database manager is not accessible (navigates to `/web/database/manager` and checks for redirect or 403)
    - `getInstalledModules()` — navigates to Apps and returns list of installed module names
    - `isModuleInstalled(moduleName: string)` — checks specific module installed status
    - `installModule(moduleName: string)` — finds and installs a module from the Apps list

    **pages/user-management.page.ts**: User CRUD via Odoo UI. Methods:
    - `openUsers()` — navigates to Settings > Users
    - `createUser(name: string, login: string, password: string, options?: { groups?: string[] })` — creates a new user via the UI form
    - `deleteUser(login: string)` — archives/deletes a user
    - `userExists(login: string)` — checks if a user exists in the users list

    **fixtures/auth.fixture.ts**: From research Pattern 3. Extend `base.test` with two custom fixtures:
    - `adminPage` — creates browser context, logs in with ADMIN_LOGIN/ADMIN_PASSWORD from env, provides authenticated page, closes context after
    - `testUserPage` — same but with TEST_USER_LOGIN/TEST_USER_PASSWORD (skip if not set)
    Export the extended `test` and re-export `expect` from `@playwright/test`.

    Key guidelines for all page objects:
    - Use `page.waitForResponse()` or wait for specific elements after Odoo form mutations (onchange RPCs)
    - Prefer `input[name="..."]`, role-based locators, and text locators over Odoo CSS classes
    - Where Odoo `o_` classes must be used, add a comment noting version-sensitivity
    - Each page object gets its own `page: Page` in constructor
    - Import types from `@playwright/test` (Page, Locator, expect)
  </action>

    - Each page file exports its class, each class has the methods described
    - `auth.fixture.ts` exports `test` and `expect`
