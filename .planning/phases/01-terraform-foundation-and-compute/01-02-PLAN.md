<task type="auto">
  <name>Task 1: Create main.tf with all DigitalOcean resource definitions</name>
  <files>infra/main.tf</files>
  <action>
Create `infra/main.tf` with the following resources, in this order:

**1. SSH Key Logic (locals + conditional resources):**
- `data "digitalocean_ssh_key" "existing"` with `count = var.use_existing_ssh_key ? 1 : 0` and `name = var.ssh_key_name`
- `resource "digitalocean_ssh_key" "uploaded"` with `count = var.use_existing_ssh_key ? 0 : 1`, `name = "${var.project_name}-deploy-key"`, `public_key = file(var.ssh_public_key_path)`
- `locals` block: `ssh_key_id = var.use_existing_ssh_key ? data.digitalocean_ssh_key.existing[0].id : digitalocean_ssh_key.uploaded[0].id`
- Follow RESEARCH.md Pattern 2 exactly

**2. VPC (IAC-01):**
- `resource "digitalocean_vpc" "main"` with `name = "${var.project_name}-vpc"`, `region = var.region`, `ip_range = var.vpc_cidr`

**3. Block Storage Volume (IAC-04):**
- `resource "digitalocean_volume" "data"` with `region = var.region`, `name = "${var.project_name}-data"`, `size = var.volume_size_gb`, `initial_filesystem_type = "ext4"`, description
- `lifecycle { prevent_destroy = true }` per user-locked decision

**4. Droplet (IAC-03):**
- `resource "digitalocean_droplet" "odoo"` with `image = "ubuntu-24-04-x64"`, `name = "${var.project_name}-odoo"`, `region = var.region`, `size = var.droplet_size`, `vpc_uuid = digitalocean_vpc.main.id`, `ssh_keys = [local.ssh_key_id]`
- `lifecycle { prevent_destroy = true }` per user-locked decision
- `connection` block: `type = "ssh"`, `user = "root"`, `private_key = file(var.ssh_private_key_path)`, `host = self.ipv4_address`, `timeout = "2m"`
- `provisioner "remote-exec"` (IAC-07) with inline commands:
  ```
  "echo 'SSH connection verified'",
  "cloud-init status --wait || true",
  "lsblk | grep -q 'disk' && echo 'Block device detected'",
  "mount | grep '/mnt/' && echo 'Volume mount verified' || echo 'WARNING: Volume not yet mounted - will be available after volume_attachment'"
  ```
  NOTE: The volume may not be mounted yet at this point because the volume_attachment resource runs after the droplet. This is expected -- the remote-exec verifies SSH connectivity and block device detection. Full mount verification happens after attachment.

**5. Volume Attachment (IAC-04):**
- `resource "digitalocean_volume_attachment" "data"` with `droplet_id = digitalocean_droplet.odoo.id`, `volume_id = digitalocean_volume.data.id`
- Use separate attachment resource (NOT inline volume_ids) per RESEARCH.md Pattern 1 and user-locked decision. This ensures correct destroy ordering (detach before delete).

**6. Cloud Firewall (IAC-02):**
- `resource "digitalocean_firewall" "main"` with `name = "${var.project_name}-fw"`, `droplet_ids = [digitalocean_droplet.odoo.id]`
- Inbound rules: SSH on port 22 from `var.allowed_ssh_ips`, HTTP on port 80 from `["0.0.0.0/0", "::/0"]`, HTTPS on port 443 from `["0.0.0.0/0", "::/0"]`
- Outbound rules: TCP all ports, UDP all ports, ICMP -- all to `["0.0.0.0/0", "::/0"]` (needed for apt, Docker pulls, etc.)
- Follow RESEARCH.md VPC + Firewall example

**Important constraints:**
- Do NOT use inline `volume_ids` on the droplet resource -- this is an anti-pattern per RESEARCH.md
- Do NOT hardcode any values that should come from variables
- Use `"${var.project_name}-"` prefix for all resource names
- Add comments referencing the requirement ID (e.g., `# IAC-01: VPC with private networking`) above each resource block
  </action>

- `grep -q 'digitalocean_volume_attachment' infra/main.tf` (separate attachment resource used)
- `grep -q 'vpc_uuid' infra/main.tf` (droplet placed in VPC)
  </verify>
