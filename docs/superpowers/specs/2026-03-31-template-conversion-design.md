# Odoo Droplet Template — Conversion Design Spec

**Date:** 2026-03-31
**Status:** Approved
**Source:** Forked from `salsedge/odoo-19.x-build` (Loodon production instance, tagged v1.0)

## Summary

Convert the Loodon-specific Odoo 19.x production deployment into a reusable GitHub template repository. New instances are created via "Use this template," producing a clean repo that an interactive setup script configures for any client/domain. A single `instance.conf` file drives all config generation. OdooKit is extracted to its own repo and referenced as a companion tool.

## Architecture

### Approach: Single Config + Shell Generator

One source of truth — `instance.conf` — defines an entire deployment. An interactive `setup.sh` populates it, and `generate.sh` reads it to produce all instance-specific config files using heredocs with variable interpolation. `.example` files serve as documentation and provide sensible defaults for demo/dev instances.

**Why this approach:**
- Zero dependencies beyond bash
- Single config file as source of truth
- Regenerable — change a value, re-run `make generate`
- `.example` files remain human-readable reference
- Conditional logic (single vs. multi-domain Nginx) handled naturally in shell

### Instance Lifecycle

```
make init          # Interactive setup → instance.conf → generate configs
make tf-init       # Initialize Terraform with generated backend
make tf-apply      # Provision DigitalOcean infrastructure
make deploy        # Full pipeline: upload → harden → docker → stack → nginx → backups
```

### Repo Distribution Model

GitHub template repository. Each new client gets their own repo via "Use this template" — clean copy, no fork history. Upstream improvements can be cherry-picked by adding the template as a remote.

## instance.conf — Configuration Surface

```bash
# --- Project Identity ---
PROJECT_NAME="odoo-demo"
ORGANIZATION="Acme Corp"

# --- Domain Configuration ---
DOMAIN_MODE="single"              # "single" or "multi"
PRIMARY_DOMAIN="odoo.example.com"
ALIAS_DOMAINS=""                  # Comma-separated, used if DOMAIN_MODE=multi
ADMIN_EMAIL="admin@example.com"

# --- DigitalOcean Infrastructure ---
DO_REGION="nyc3"
DROPLET_SIZE="s-2vcpu-4gb"
VOLUME_SIZE_GB="25"
SSH_KEY_FINGERPRINT=""            # Blank = upload ~/.ssh/id_ed25519.pub
SSH_PORT="9292"

# --- Database ---
DB_NAME="odoo-01"
DB_USER="odoo"

# --- Spaces (Object Storage) ---
TFSTATE_BUCKET="${PROJECT_NAME}-tfstate"
BACKUP_BUCKET="${PROJECT_NAME}-backups"
SPACES_REGION="${DO_REGION}"

# --- Backup ---
BACKUP_RETENTION_DAYS="30"
BACKUP_NOTIFY_EMAIL="${ADMIN_EMAIL}"

# --- Monitoring (optional) ---
ICINGA2_ENABLED="false"
ICINGA2_MASTER=""

# --- OdooKit (optional) ---
ODOOKIT_REPO="https://github.com/salsedge/odookit"
```

## setup.sh — Interactive Setup

### Flow

Walks through `instance.conf.example` section by section. Each value shows its default; operator presses Enter to accept or types an override.

### Key Behaviors

- **Cascading defaults:** `ADMIN_EMAIL` feeds into `BACKUP_NOTIFY_EMAIL`. `PROJECT_NAME` feeds into bucket names. Change the root value, downstream defaults update automatically.
- **Validation:** Domain format check, DO region must be valid, port must be numeric, bucket names must be DNS-compliant.
- **Idempotent:** Running again reads existing `instance.conf` as new defaults — tweak one value without re-entering everything.
- **Auto-generated secrets:** Generates random passwords for DB and Odoo admin master password, writes to `config/.env`. Operator never types passwords unless they want to override.
- **Bucket creation:** Uses `s3cmd` or `doctl` (whichever is available) to create the tfstate Spaces bucket. Fails gracefully if bucket already exists.
- **Review step:** Shows full summary of all values before writing. Operator confirms before anything is written.

### Output

Writes `instance.conf`, then calls `generate.sh` to produce all config files.

## generate.sh — Config Generation

Reads `instance.conf`, produces all instance-specific files using heredocs with variable interpolation.

### Generated Files

| Generated File | Key Values Injected |
|---|---|
| `infra/backend.tf` | `TFSTATE_BUCKET`, `SPACES_REGION` |
| `infra/terraform.tfvars` | `PROJECT_NAME`, `DO_REGION`, `DROPLET_SIZE`, `VOLUME_SIZE_GB`, `SSH_KEY_FINGERPRINT`, `SSH_PORT` |
| `config/.env` | `DB_NAME`, `DB_USER`, DB passwords (random), Odoo admin password (random) |
| `config/odoo.conf` | `DB_NAME`, `DB_USER`, admin password token |
| `config/nginx/odoo-pre-ssl.conf` | `PRIMARY_DOMAIN` |
| `config/nginx/odoo.conf` | `PRIMARY_DOMAIN`, `ALIAS_DOMAINS`, `DOMAIN_MODE` (branches to single or multi template) |
| `config/rclone.conf` | `BACKUP_BUCKET`, `SPACES_REGION` |
| `config/msmtprc` | `ADMIN_EMAIL` |
| `monitoring/icinga2/commands.conf` | `DB_NAME`, `DB_USER` (only if `ICINGA2_ENABLED=true`) |

### Key Behaviors

- **Password preservation:** If `config/.env` already exists, preserves existing passwords and only updates non-secret values. Prevents accidentally rotating credentials on re-generate.
- **Nginx branching:** `DOMAIN_MODE=single` generates straightforward single-domain config. `DOMAIN_MODE=multi` generates server block with all alias domains plus redirect block.
- **Feature gating:** If `ICINGA2_ENABLED=false`, skips monitoring config generation.
- **Generated file headers:** Each file gets `# Generated from instance.conf — do not edit directly. Re-run: make generate` comment header.
- **Re-runnable:** Change a value in `instance.conf`, run `make generate`, all configs update (respecting password preservation).

## Makefile Targets

```makefile
# --- Setup ---
init              # Interactive setup → instance.conf → generate all configs
generate          # Re-generate configs from instance.conf (no prompts)
validate          # Pre-flight: instance.conf complete, configs exist, bucket reachable

# --- Infrastructure ---
tf-init           # terraform init with generated backend
tf-plan           # terraform plan
tf-apply          # terraform apply
tf-destroy        # terraform destroy (with confirmation)

# --- Deployment ---
deploy            # Full pipeline: upload → harden → docker → stack → nginx → backups
upload            # SCP configs + scripts to droplet
deploy-phase1     # 01-harden-host.sh (reconnects on new SSH port)
deploy-phase2     # 02-install-docker.sh
deploy-phase3     # 03-deploy-stack.sh
deploy-phase4     # 04-setup-nginx.sh (uses DOMAIN_MODE for config selection)
deploy-phase5     # 05-setup-backups.sh

# --- Day-2 Operations ---
set-domain        # Add/change domains post-deploy
ssh               # SSH to instance (IP from terraform output, port from instance.conf)
status            # Remote health check
logs              # Tail Odoo/PG logs
backup-now        # Trigger manual backup
test              # Delegates to OdooKit if cloned, else prints install instructions

# --- Cleanup ---
clean             # Remove generated files (keeps instance.conf)
nuke              # Remove everything including instance.conf (fresh start)
```

SSH connection details derived automatically — IP from `terraform output`, port from `instance.conf`.

## OdooKit Extraction

### In This Repo (Template)

- `odookit/` directory removed entirely
- `odookit/` added to `.gitignore` (it's a separate repo cloned in)
- `README.md` includes "Recommended Companion Tools" section pointing to OdooKit repo
- `make init` asks "Clone OdooKit into project? (y/n)" — if yes, clones into `odookit/` and runs its setup (reads from `instance.conf` for domain/credentials)
- `make test` delegates to OdooKit if the directory exists, otherwise prints install instructions

### OdooKit as Its Own Repo

Separate effort — not part of this design. The template assumes OdooKit exists at `salsedge/odookit` and can read `../instance.conf` or accept URL + credentials directly.

## Nginx Domain Modes

### Single Domain Mode

Standard config: one `server_name`, SSL termination, proxy to Odoo upstream.

### Multi-Domain Mode

Primary domain served directly. Each alias domain gets a `server` block that 301-redirects to the primary. All domains listed on the SSL certificate (certbot `--expand` with `-d` for each).

### Post-Deploy Domain Changes

`make set-domain` (existing `scripts/ops/set-domain.sh`) handles adding/changing domains on a live instance. Updates Nginx config and expands the SSL certificate.

## Definition of "Done" After Setup

After `make init` → `make tf-apply` → `make deploy`, the instance has:

1. Hardened Ubuntu host (SSH key-only, non-default port, UFW, fail2ban, auditd)
2. Docker CE with `iptables: false`
3. Running Odoo 19 + PostgreSQL 18 on Docker Compose
4. Nginx reverse proxy with Let's Encrypt SSL (single or multi-domain)
5. Automated daily backups with offsite sync to DO Spaces
6. First backup verified

Production-ready. No manual follow-up steps required.

## .example File Convention

Current instance-specific config files (e.g., `infra/backend.tf`, `config/odoo.conf`) are renamed to `.example` extensions (e.g., `infra/backend.tf.example`, `config/odoo.conf.example`). These `.example` files are committed to the repo as human-readable references with sensible demo defaults. The actual config files are produced by `generate.sh` and are gitignored. Nginx configs follow the same pattern: the current `config/nginx/odoo.conf` and `odoo-pre-ssl.conf` are replaced by `odoo-single.conf.example` and `odoo-multi.conf.example` as references, with `generate.sh` producing the actual `odoo.conf` and `odoo-pre-ssl.conf`.

## Out of Scope

- OdooKit repo creation (separate effort)
- Kubernetes, Docker Swarm, or multi-droplet architectures
- Prometheus/Grafana (Icinga2 is the monitoring stack)
- Terraform workspaces for multi-instance in a single repo
- Odoo module development or CRM buildout
- WireGuard VPN (deferred to v2 of the template)
